## Раздел 1: Описание проблемы

### Что такое Race Condition?

Race condition (состояние гонки) — ситуация, когда результат зависит от порядка выполнения параллельных операций. В БД это возникает, когда несколько транзакций одновременно читают и изменяют одни данные.

**Пример из жизни:**
- Пользователь дважды кликнул "Оплатить"
- Два микросервиса параллельно обрабатывают один платёж
- Сеть дала сбой, запрос отправился повторно

### Почему READ COMMITTED не защищает?

На READ COMMITTED:
1. Каждая транзакция видит закоммиченные данные на момент запроса
2. Две транзакции одновременно читают статус 'created'
3. Обе проходят проверку и выполняют UPDATE
4. Результат: двойная оплата

**Демонстрация проблемы:**

```
Время | Сессия 1                    | Сессия 2
------|----------------------------|---------------------------
t1    | BEGIN                      |
t2    | SELECT status (created)    |
t3    |                            | BEGIN
t4    |                            | SELECT status (created)
t5    | UPDATE status = paid       |
t6    |                            | UPDATE status = paid (ждёт)
t7    | COMMIT                     |
t8    |                            | UPDATE выполняется
t9    |                            | COMMIT -> ДВЕ ЗАПИСИ!
```

### Примеры из реальной жизни

1. **Двойной клик на "Оплатить"**
   - Пользователь нажал дважды
   - Два HTTP-запроса приходят одновременно
   - Оба начинают обработку параллельно

2. **Микросервисная архитектура**
   - Несколько реплик сервиса оплаты
   - Два инстанса получают запрос на один заказ
   - Оба видят статус 'created' и списывают средства

3. **Сбой сети**
   - Клиент не получил ответ из-за таймаута
   - Пользователь нажимает "Оплатить" снова
   - Первый запрос уже обрабатывается

## Раздел 2: Уровни изоляции в PostgreSQL

### READ UNCOMMITTED

**Описание:** Чтение незакоммиченных данных (dirty reads).

**Предотвращает:** Ничего

**Не предотвращает:**
- Dirty reads
- Non-repeatable reads
- Phantom reads

**Когда использовать:** Практически никогда.

**Особенность в PostgreSQL:** Работает как READ COMMITTED из-за MVCC.

---

### READ COMMITTED (по умолчанию)

**Описание:** Каждый SELECT видит данные, закоммиченные на момент начала запроса.

**Предотвращает:**
- ✅ Dirty reads

**Не предотвращает:**
- ❌ Non-repeatable reads
- ❌ Phantom reads

**Пример non-repeatable read:**
```sql
-- Сессия 1
BEGIN;
SELECT balance FROM accounts WHERE id = 1; -- 1000
-- Сессия 2 изменяет balance и делает COMMIT
SELECT balance FROM accounts WHERE id = 1; -- 500 (!)
COMMIT;
```

**Когда использовать:** Большинство приложений, обычные операции.

---

### REPEATABLE READ

**Описание:** Транзакция видит snapshot данных на момент первого запроса.

**Предотвращает:**
- ✅ Dirty reads
- ✅ Non-repeatable reads
- ✅ Phantom reads (в PostgreSQL)

**Не предотвращает:**
- ❌ Write skew

**Когда использовать:** Финансовые операции, сложные отчёты.

---

### SERIALIZABLE

**Описание:** Транзакции выполняются как последовательные, даже если параллельны физически.

**Предотвращает:**
- ✅ Все аномалии

**Недостатки:**
- ❌ Откаты транзакций при конфликтах
- ❌ Снижение производительности
- ❌ Требует retry logic

**Когда использовать:** Критичные операции: переводы, бронирования, аукционы.

---

### Сравнительная таблица

| Уровень изоляции  | Dirty Read | Non-Repeatable Read | Phantom Read | Performance |
|-------------------|------------|---------------------|--------------|-------------|
| READ UNCOMMITTED  | ❌          | ❌                   | ❌            | Высокая     |
| READ COMMITTED    | ✅          | ❌                   | ❌            | Высокая     |
| REPEATABLE READ   | ✅          | ✅                   | ✅            | Средняя     |
| SERIALIZABLE      | ✅          | ✅                   | ✅            | Низкая      |

## Раздел 3: Решение проблемы

### Почему REPEATABLE READ решает проблему?

REPEATABLE READ использует snapshot isolation:
1. Транзакция видит снапшот на момент первого запроса
2. Изменения других транзакций не видны
3. При конфликте UPDATE — ошибка сериализации

Но без FOR UPDATE проблема остаётся — обе транзакции читают 'created' в своих снапшотах.

### Зачем нужен FOR UPDATE?

`FOR UPDATE` создаёт эксклюзивную блокировку строки:

**Без FOR UPDATE:**
```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT status FROM orders WHERE id = '...';
-- Другая транзакция может читать ту же строку
UPDATE orders SET status = 'paid' WHERE id = '...';
COMMIT;
```

**С FOR UPDATE:**
```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT status FROM orders WHERE id = '...' FOR UPDATE;
-- Другая транзакция ЖДЁТ блокировки
UPDATE orders SET status = 'paid' WHERE id = '...';
COMMIT;
```

**Типы блокировок:**
- `FOR UPDATE` — эксклюзивная (никто не может изменить)
- `FOR SHARE` — разделяемая (можно читать, нельзя менять)
- `FOR NO KEY UPDATE` — слабее FOR UPDATE
- `FOR KEY SHARE` — минимальная блокировка

### Что без FOR UPDATE на REPEATABLE READ?

Обе транзакции читают 'created' в своих снапшотах. При UPDATE второй — ошибка сериализации.

Но:
- Требуется обработка ошибок
- Лишние попытки UPDATE
- При нагрузке много откатов

### Разница FOR UPDATE и FOR SHARE

| Характеристика      | FOR UPDATE | FOR SHARE |
|---------------------|------------|-----------|
| Тип блокировки      | Эксклюзивная | Разделяемая |
| Блокирует FOR UPDATE| ✅         | ✅         |
| Блокирует FOR SHARE | ✅         | ❌         |
| Блокирует UPDATE    | ✅         | ✅         |
| Use case            | Перед изменением | Защита от изменений |

## Раздел 4: Рекомендации для продакшена

### Какой ISOLATION LEVEL использовать?

**Рекомендация:** READ COMMITTED как default + REPEATABLE READ + FOR UPDATE для критичных операций.

#### Обоснование:

**1. Производительность**
- READ COMMITTED — минимальный overhead
- Блокировки удерживаются минимальное время
- Нет массових откатов как в SERIALIZABLE

**2. Безопасность данных**
- Критичные операции защищены FOR UPDATE
- Некритичные работают на READ COMMITTED
- Чёткое разделение ответственности

**3. Риски deadlock**
- FOR UPDATE может вызвать deadlock
- Решение: блокировать ресурсы в одном порядке
- Пример: order → order_items → payment

**4. Простота разработки**
- READ COMMITTED понятен и легко отлаживается
- REPEATABLE READ + FOR UPDATE явно помечает критичные места
- Код самодокументируется

**5. Масштабируемость**
- READ COMMITTED хорошо масштабируется
- Локальные блокировки не создают bottleneck
- Connection pooling работает эффективно

### Итоговая рекомендация

1. **Default:** READ COMMITTED — 95% операций
2. **REPEATABLE READ + FOR UPDATE:** оплата, баланс, резервирование — 5%
3. **Optimistic locking:** счётчики, рейтинги
